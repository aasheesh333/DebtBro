   - Today, Jul 3 7:13 PM, Redmi Note 6 Pro / Android 9, app v1.0(144): `IllegalStateException: TypeToken must be created with a type argument` at `SplitViewModel.java:144`
   - 3 days ago, Jun 30 9:38 AM, same model: `IllegalArgumentException: Invalid column data1` at `SplitScreen.kt:328`

3. AI roast always returns Hinglish even when user selected non-Hindi language on onboarding page 0. Root cause: `AiRepository.kt:319-358` `when(roastLevel)` block hardcodes "Use Hinglish naturally" + Indian-scenario phrases for all 3 roast levels regardless of `selectedLangCode` — the existing `langInstruction` prefix at `:292-313` is correct but is overridden by contradictory prompt body.

4. Onboarding advertises **25 languages** in `LanguageSelectorGrid.kt:32-58`, but only **10** have any translations in `LocalizedString.kt` (en/hi/es/fr/de/ja/mr/pa/gu/bn — and pa/gu/bn are 29-36 entry skeletons vs en's 238). 15 advertised languages silently fall back to English. Plus ~31 hardcoded English UI strings across `AddDebtBottomSheet.kt`, `DebtDetailScreen.kt`, `SettingsScreen.kt`, `SplitScreen.kt`, `AuthScreen.kt`, various ViewModels that bypass `LocalizedString.get()` entirely.

User decisions:
- Onboarding approach: **Approach A** — split current page 4 into 2 separate pages (name → auth), 6 onboarding pages total.
- Crash fix stance: defensive code + ProGuard keep rules + runCatching wrap on launches.
- AI gating stance: separate codepath for `isIndicFamily` (hi/mr/pa/gu); language-neutral tone for everyone else.
- i18n scope: **Scope A — Full coverage of all 25 advertised languages**. Translations generated by opencode via model `nvidia/z-ai/glm-5.2`.

## Section 2 — Split bill crash fixes

### Crash #1 — `SplitViewModel.kt:144` — TypeToken+R8

**Stack trace (Crashlytics, issue 594a7f30...):**
```
com.google.gson.reflect.TypeToken.getTypeTokenTypeArgument (TypeToken.java:100)
com.google.gson.reflect.TypeToken.<init> (TypeToken.java:70)
com.dhanuk.debtbro.presentation.screens.split.SplitViewModel$getAiSummary$1$names$1.<init> (SplitViewModel.java:144)
com.dhanuk.debtbro...SplitViewModel$createSplit$1.invokeSuspend (SplitViewModel.kt:118)
```

**Root cause:** `object : TypeToken<List<String>>() {}.type` is an anonymous subclass whose generic type argument is stripped by R8 in release builds. At runtime the class has no type argument → `TypeToken.getTypeTokenTypeArgument` throws `IllegalStateException`.

**Two sites in `SplitViewModel.kt` use this pattern:**
1. Lines 123-126 (`createDebtsFromSplit`): `Gson().fromJson(split.participants, object : TypeToken<List<String>>() {}.type)`
2. Lines 142-145 (`getAiSummary`): same pattern

### Fix Crash #1 — code (two layers)

**Layer 1 — code-level (primary):** replace the anonymous `TypeToken` subclass with the explicit parameterized form that doesn't depend on class-signature retention:
```kotlin
import com.google.gson.reflect.TypeToken
import java.lang.reflect.Type

private val LIST_STRING_TYPE: Type =
    TypeToken.getParameterized(List::class.java, String::class.java).type

// usages:
Gson().fromJson(split.participants, LIST_STRING_TYPE)
```
`TypeToken.getParameterized(...)` builds the `Type` from explicit class objects — no anonymous subclass, no signature-retention dependency. Hoisted to a `private val` so it's constructed once.

**Layer 2 — ProGuard keep rules (defense-in-depth, second layer):** append to `app/proguard-rules.pro`:
```
-keep class com.google.gson.** { *; }
-keepattributes Signature
-keepattributes EnclosingMethod
-keepattributes InnerClasses
-keep class * implements com.google.gson.TypeAdapterFactory
-keep class * implements com.google.gson.JsonSerializer
-keep class * implements com.google.gson.JsonDeserializer
```
`-keepattributes Signature` ensures generic signatures survive even where anonymous TypeTokens appear in future code.

### Crash #2 — `SplitScreen.kt:328` — Contacts cursor `data1`

**Stack trace (Crashlytics, issue 94720452...):**
```
android.database.DatabaseUtils.readExceptionFromParcel (DatabaseUtils.java:165)
android.content.ContentResolver.query (ContentResolver.java:711)
com.dhanuk.debtbro.presentation.screens.split.SplitScreenKt.ContactPickerBottomSheet$lambda$19 (SplitScreen.kt:328)
```

**Root cause:** `ContactPickerBottomSheet` lambda queries a `ContactsContract` URI with a projection array containing literal `"data1"`. When the URI doesn't correspond to a MIME type that exposes `data1` (e.g., querying `Contacts.CONTENT_URI` instead of `CommonDataKinds.Phone.CONTENT_URI`), the Binder receives an IllegalArgumentException from the content provider inside `readExceptionFromParcel`. The `column data1` is invalid for that row.

### Fix Crash #2 — ContactPickerBottomSheet rewrite
- Use `ContactsContract.CommonDataKinds.Phone.CONTENT_URI` as the query URI (instead of generic Contacts URI) — this is the contract for phone numbers where `data1` IS the phone number column.
- Replace literal `"data1"` with `ContactsContract.CommonDataKinds.Phone.NUMBER` (which is `data1` semantically and equal to the string "data1" — but using the contract constant guarantees the URI+column pairing matches).
- Wrap the query in `runCatching { contentResolver.query(...) }.getOrNull()` so IPC failures become null cursor instead of an exception.
- Defensive column access: `cursor.getColumnIndex(ContactsContract.CommonDataKinds.Phone.NUMBER).takeIf { it >= 0 }?.let { cursor.getString(it) }` — bail to "contact has no phone number" silently if column missing.
- Wrap entire getter in `try { } catch (e: SecurityException) { /* ignore — permission revoked */ }` to handle `READ_CONTACTS` revocation.

### Fix Crash #3 — defensive wrap of SplitViewModel.launches

Even after the targeted fixes, wrap the 3 `viewModelScope.launch` blocks in `SplitViewModel.kt` with `runCatching` + Snackbar error UI so any future exception in the split flow surfaces to user as "Couldn't create split: ${message}" instead of silent no-op or a crash:
- `createSplit` (`SplitViewModel.kt:93-120`) — wrap the body inside `viewModelScope.launch { runCatching { ... }.onFailure { snackbar.emit("Couldn't create split: ${it.message}") } }`
- `createDebtsFromSplit` (`SplitViewModel.kt:122-139`) — same pattern.
- `getAiSummary` (`SplitViewModel.kt:141-149`) — same pattern, error → "Couldn't fetch split summary".

Need a `SharedFlow<String> snackbar` exposed to `SplitScreen` for rendering. Pattern mirrors `SettingsViewModel` and `AnalyticsViewModel` snackbar usage.

### Code impact
- `SplitViewModel.kt`: Replace 2 TypeToken anonymous class sites with `LIST_STRING_TYPE` constant. Wrap 3 launch bodies. Add `snackbar` SharedFlow. ~30 lines churn.
- `SplitScreen.kt`: Rewrite `ContactPickerBottomSheet` lambda. ~30 lines.
- `app/proguard-rules.pro`: Append ~6 lines Gson keep rules.

## Section 3 — AI roast language gating

### Root cause (precise, line-level)
`AiRepository.kt:291-358` `buildSystemPrompt(roastLevel, langCode, debtType)` returns `$langInstruction\n$prompt"` — the leading `langInstruction` line correctly says e.g., "Respond ONLY in Spanish." Then the `when(roastLevel)` block at `:319-357` builds the `prompt` body with hardcoded "Use Hinglish naturally" + Indian-culture references in all 3 roast levels:

- **MILD** (`:320-330`): `"You are a witty Indian friend writing a WhatsApp message about money." … "- Use Hinglish naturally (mix Hindi and English)" "- Be warm, funny, and creative — use metaphors or shared jokes"`
- **SPICY** (`:331-346`): `"You are a brutally funny Indian debt collector with legendary comedic timing." … "- Use Hinglish naturally" "- Be creatively savage — use wild metaphors or Bollywood comparisons"`
- **MEDIUM (else)** (`:347-357`): `"You are a clever, sarcastic Indian friend dropping a subtle money hint." … "- Use Hinglish naturally" "- Use relatable Indian scenarios (chai, zomato, petrol prices)"`

Additionally `analyzeDebts` (`:421`) user-prompt: `"Give ONE sharp, funny, honest 2-line insight. Hinglish welcome."` and `generateSplitSummary` (`:446`): `"Write ONE funny line about this. Hinglish ok."` — both hardcode Hinglish too.

Gemini honors the leading "Respond ONLY in Spanish" instruction, but the conflicting "Use Hinglish naturally" cues leak Hindi phrases into the response.

### Fix design
Refactor into a `buildRoastPersona(roastLevel, langCode)` helper that branches on `isIndicFamily`:

```kotlin
private fun isIndicFamily(langCode: String): Boolean =
    langCode in setOf("hi", "mr", "pa", "gu")

private fun buildRoastPersona(roastLevel: RoastLevel, langCode: String): String {
    val indic = isIndicFamily(langCode)
    val (baseRole, scenarios, savageLine) = when (roastLevel) {
        RoastLevel.MILD -> Triple(
            "a witty friend writing a WhatsApp message about money",
            "warm, funny, creative — metaphors or shared jokes",
            null as String?
        )
        RoastLevel.MEDIUM -> Triple(
            "a clever, sarcastic friend dropping a subtle money hint",
            "relatable everyday scenarios",
            null
        )
        RoastLevel.SPICY -> Triple(
            "a brutally funny debt collector with legendary comedic timing",
            "creatively savage — wild metaphors" + (if (indic) " or Bollywood comparisons" else " or pop-culture references"),
            null
        )
    }
    val rolePrefix = if (indic) "an Indian $baseRole" else "a $baseRole"
    val lingual = if (indic) {
        "Use Hinglish naturally (mix Hindi and English). Use relatable Indian scenarios (chai, zomato, petrol prices)."
    } else {
        "Use your selected language naturally. Use relatable everyday scenarios."
    }
    return "You are $rolePrefix. Be $scenarios. $lingual"
}
```

`buildSystemPrompt(roastLevel, langCode, debtType)` then becomes:
```kotlin
private fun buildSystemPrompt(roastLevel: RoastLevel, langCode: String, debtType: String?): String {
    val langInstruction = when(langCode) { /* ... existing logic unchanged ... */ }
    val persona = buildRoastPersona(roastLevel, langCode)
    return "$langInstruction\n$persona"
}
```

For `analyzeDebts` (`:421`) and `generateSplitSummary` (`:446`), gate the trailing phrase:
```kotlin
val lingualHint = if (isIndicFamily(langCode)) "Hinglish welcome" else "Use your selected language"
// user prompt = "Give ONE sharp, funny, honest 2-line insight. $lingualHint."
// user prompt = "Write ONE funny line about this. $lingualHint."
```

### Code impact
- `AiRepository.kt:291-358`: refactor `when(roastLevel)` block — extract to `buildRoastPersona` helper. ~30 lines churn.
- `AiRepository.kt:421, :446`: gate "Hinglish welcome/ok" with `isIndicFamily`. ~4 lines.
- No new files, no new pref, no new dependencies.
